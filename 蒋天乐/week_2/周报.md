## 二、 每日学习进度与详实内容

### 星期一：IDEA高阶分支管理与图形化操作
*   **多分支可视化切换**：学习了在IDEA右下角状态栏快速调出Git Branches面板 。熟练掌握了对本地分支和远程分支（`Remote Branches`）进行 `Checkout`（切换）、`New Branch from Selected`（基于指定版本拉取分支）、`Compare with Current`（与当前分支对比差异）等高频操作 。
*   **Version Control 面板深度运用**：利用IDEA底部的 Version Control (或 Git) 面板，通过 `Log` 选项卡清晰查看带有不同颜色分支线（Branch Graph）的提交历史 。
*   **右键菜单的高效操作**：掌握了在代码编辑区或项目树中右键选择 `Git`，快速执行 `Add`、`Commit`、`Push`、`Pull`、`Show History`（查看单个文件历史）、`Revert`（还原未提交修改）等日常操作，摆脱繁琐的纯命令行记忆 。

### 星期二：攻克合并冲突（IDEA可视化解决实战）
*   **冲突场景再现**：当执行 `merge` 或 `pull` 操作时，若本地分支与目标分支（或远程分支）修改了同一文件的同一行，IDEA会自动弹出 `Files Merged with Conflicts` 警告框 。
*   **三栏式冲突解决神器**：
    *   点击弹窗中的 `Merge...` 按钮，进入IDEA强大的三栏对比界面 。
    *   **左侧（Local/Yours）**：代表当前自己本地分支的代码 。
    *   **右侧（Server/Theirs）**：代表将被合并进来的外部/远程代码 。
    *   **中间（Result）**：代表最终解决后的合并结果 。
    *   通过点击中间栏两侧的 `>>` 或 `<<` 箭头（或者选择 `Accept Yours` / `Accept Theirs`），能够直观地采纳或舍弃特定代码块，最后点击 `Apply` 自动完成冲突文件的清理并加入暂存区 。

### 星期三：企业级团队协作沙盘演练（上半场：建库与认领）
*   **场景设定**：模拟了一个真实的开发团队，包含一名“项目经理（组长）”和两名“组员（A和B）” 。
*   **组长动作（搭建脚手架）**：
    *   项目经理负责在本地初始化项目脚手架（如 SpringMVC Demo）并创建 `.gitignore` 文件 。
    *   在码云（Gitee）创建公共远程仓库，并将本地项目通过 `git push` 推送至远程仓库的主分支 。
*   **组员动作（加入开发）**：
    *   组员A和组员B分别通过 `git clone` 将远程仓库的项目克隆到各自的IDEA工作区中，形成各自独立的工作副本，正式开始并行开发 。

### 星期四：企业级团队协作沙盘演练（下半场：冲突与同步）
*   **正常提交流程**：组员A在自己的工作区修改了文件，依次执行 `commit` 和 `push`，顺利将代码推送至远程仓库 。
*   **多人协作冲突触发**：组员B在同一时间段内，碰巧修改了**同一个文件的同一行代码** 。组员B在本地提交（`commit`）成功，但在尝试 `push` 到远程仓库时，由于远程仓库已被组员A更新，导致推送被拒绝（Rejected） 。
*   **化解危机（先拉后推）**：
    *   组员B遵循规范，立刻执行 `git pull` 获取远程仓库（组员A的提交）的最新代码 。
    *   `pull` 操作触发本地合并冲突，组员B利用IDEA的冲突解决工具手动保留双方的有效代码 。
    *   冲突解决后，组员B重新生成了一个合并提交（Merge Commit），再次执行 `git push`，成功将包含A和B两人工作成果的代码同步至云端 。

### 星期五：Git日常铁令、疑难杂症排查与课程模型适配
*   **Git 开发三条铁令**：
    1.  **切换分支前先提交**：工作区有未提交的修改时切分支极易导致代码串区或丢失，必须先 `commit` 。
    2.  **代码及时提交**：只要代码被 `commit` 到本地仓库，就有了历史记录，几乎永远不会彻底丢失（可通过reflog找回） 。
    3.  **遇事莫删目录**：遇到任何Git报错或难以解决的冲突，**绝对不要**通过直接删除 `.git` 文件夹或项目目录来逃避，应寻求正确的回退或合并指令 。
*   **IDEA集成Git Bash（终端增强）**：为统一命令行体验，在IDEA `Settings -> Tools -> Terminal -> Shell path` 中将默认的cmd/powershell替换为 `C:\Program Files\Gitinash.exe`，实现直接在IDEA底部窗口执行原生Git命令 。
*   **教学/实战分支模型适配**：了解了后续项目的代码分发模式。线上包含 `book/xx` 分支（标准代码），日常开发需执行 `fetch` 后，从对应的 `book` 节点创建自己的 `dev/xx` 分支进行学习和修改，互不干扰 。

## 三、 本周学习疑难问题与解决方案
1.  **Windows下无法右键直接创建 `.gitignore` 文件** ：
    *   *问题描述*：Windows资源管理器有时不允许创建以 `.` 开头且无文件名的文件 。
    *   *解决办法*：熟练使用Git Bash，在项目目录下通过执行 `touch .gitignore` 命令即可顺利创建隐藏文件 。若需查看，需在资源管理器“查看”选项卡中勾选“隐藏的项目” 。
2.  **IDEA中修改了文件却没有变色，也无法提交**：
    *   *原因剖析*：文件可能处于 `.gitignore` 的排除规则范围内，或者整个项目并未正确纳入版本控制（Unversioned Files）。
    *   *解决办法*：检查忽略文件规则；如果在Unversioned Files组中，需右键执行 `Git -> Add` 手动追踪。
3.  **对 "pull" 操作的底层逻辑理解模糊**：
    *   *原因剖析*：总是分不清 `fetch` 和 `pull`。
    *   *解决办法*：通过画图彻底记牢公式：`git pull = git fetch + git merge` 。`pull` 不仅下载远程更新，还会尝试将远程更新融合到当前分支工作区 ；若想“只看不合”，则应使用 `fetch` 。
