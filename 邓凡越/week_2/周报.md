## 二、核心学习内容

### 2.1 PageHelper 分页插件原理

PageHelper 是基于 MyBatis 拦截器机制实现的分页插件，通过拦截 Executor 的 query 方法，在执行 SQL 前自动拼接分页语句，实现了对业务代码的零侵入。

**核心特性**：
- **零侵入**：无需修改已有 SQL，只需在查询前调用 `PageHelper.startPage()` 即可
- **线程安全**：分页参数通过 `ThreadLocal` 传递，自动清理
- **多数据库支持**：自动识别数据库方言，生成对应的分页 SQL

### 2.2 分页查询完整实现

#### 2.2.1 Service 层

Service 层负责接收分页参数，调用 PageHelper 启动分页，并返回分页结果对象。

**Service 层**

```java
@Service
public class UserService {
    @Autowired
    private UserMapper userMapper;

    public PageInfo<User> listUsers(Integer pageNum, Integer pageSize, String keyword) {
        PageHelper.startPage(pageNum, pageSize);
        List<User> list = userMapper.selectByKeyword(keyword);
        return new PageInfo<>(list);
    }
}
```

**关键点说明**：
- `PageHelper.startPage(pageNum, pageSize)` 必须在 Mapper 查询之前调用，且仅对紧跟其后的一条查询生效
- `PageInfo<T>` 封装了分页结果，包含 `total`（总记录数）、`pages`（总页数）、`list`（当前页数据）等字段

#### 2.2.2 Mapper 接口

**Mapper 接口**

```java
@Mapper
public interface UserMapper {
    List<User> selectByKeyword(String keyword);
}
```

#### 2.2.3 Mapper XML

Mapper XML 中实现带条件判断的动态 SQL，配合 PageHelper 自动完成分页。

**Mapper XML**

```xml
<select id="selectByKeyword" resultType="User">
    SELECT * FROM user
    <where>
        <if test="keyword != null and keyword != ''">
            username LIKE CONCAT('%', #{keyword}, '%')
        </if>
    </where>
</select>
```

**技术要点**：
- `<where>` 标签自动处理开头的 `AND`/`OR`，且在没有条件时不生成 `WHERE` 子句
- `CONCAT` 函数拼接模糊查询的通配符，实现关键字搜索

#### 2.2.4 Controller 层

**Controller 层**

```java
@RestController
@RequestMapping("/users")
public class UserController {
    @Autowired
    private UserService userService;

    @GetMapping
    public Result listUsers(
            @RequestParam(defaultValue = "1") Integer pageNum,
            @RequestParam(defaultValue = "10") Integer pageSize,
            @RequestParam(required = false) String keyword) {
        PageInfo<User> pageInfo = userService.listUsers(pageNum, pageSize, keyword);
        return Result.success(pageInfo);
    }
}
```

**设计要点**：
- `defaultValue` 设置分页默认值：第一页、每页 10 条
- `required = false` 使关键字参数可选，支持不带搜索条件查询全量数据
- 统一返回封装类 `Result` 保证接口响应格式一致

### 2.3 进阶配置

#### 2.3.1 带排序的分页查询

```java
PageHelper.startPage(pageNum, pageSize, "id DESC");
// 或者
PageHelper.startPage(pageNum, pageSize);
PageHelper.orderBy("id DESC");
```

`orderBy` 参数直接写入 SQL 的 `ORDER BY` 子句，避免前端拼接排序字段时的 SQL 注入风险。

#### 2.3.2 不查总记录数（提升性能）

```java
PageHelper.startPage(pageNum, pageSize, false);
```

当数据量较大或仅需展示数据而无需总页数时，可跳过 `COUNT` 查询，显著提升查询性能。

#### 2.3.3 前端对接的 JSON 格式

**前端拿到的 JSON**

```json
{
  "code": 200,
  "data": {
    "pageNum": 1,
    "pageSize": 10,
    "total": 100,
    "pages": 10,
    "list": [
      { "id": 1, "username": "zhangsan" },
      { "id": 2, "username": "lisi" }
    ]
  }
}
```

**PageInfo 核心字段说明**：

| 字段     | 说明              |
| -------- | ----------------- |
| pageNum  | 当前页码          |
| pageSize | 每页显示条数      |
| total    | 总记录数          |
| pages    | 总页数            |
| list     | 当前页数据集合    |

## 四、问题与解决方案

### 4.1 PageHelper 分页不生效

**问题**：在 Mapper 查询后执行了其他查询操作，导致 PageHelper 分页参数被后续查询消费，目标查询未分页。

**解决**：确保 `PageHelper.startPage()` 与目标 Mapper 查询方法之间无其他数据库查询操作；如需多次分页查询，每次查询前重新调用 `startPage()`。

### 4.2 分页总数查询性能瓶颈

**问题**：大表查询时 COUNT 操作耗时过长，影响接口响应速度。

**解决**：对于不需要展示总页数的场景（如滚动加载），使用 `PageHelper.startPage(pageNum, pageSize, false)` 跳过 COUNT 查询，直接返回指定页的数据。
